🧠 核心技术 ①：LLM Function Calling

DeepSeek 自主决策工具调用

将 6 个 LangChain Tool 转为 OpenAI 兼容的函数定义，LLM 根据用户意图自主决定调用哪些工具、以什么顺序、传什么参数

🔄 Function Calling 循环

用户消息 → LLM 分析 → tool_calls[] → 执行工具 → 结果反馈 LLM → 生成报告

⚡ 最多 20 轮迭代，temperature=0 保证确定性，自动防无限循环

📋 6 个函数定义：

🔍 query_contract_url
  → requestId → 附件列表
📥 download_contract
  → file_id → 本地文件
🖼️ prepare_images
  → PDF → 图片列表
🔤 ocr_images
  → 图片 → 识别文字
💰 extract_amounts
  → 文字 → 金额 JSON
✅ validate_amounts
  → 金额 → 校验结果
Function Calling

🔤 核心技术 ②：多 OCR 引擎适配层

策略模式 + 统一接口

通过 `ocr/` 子包抽象统一的 OCR 接口，4 种引擎各自实现，config.yaml 一键切换

引擎类型适用场景
🔷 MCP远程 SSE内网 MCP 网关，零配置
🔶 百度云 API通用文字识别，中文优化
🟣 阿里云云 API印刷体识别，高精度
🔵 腾讯云云 API通用印刷体，腾讯生态

📋 配置示例：

ocr:
  provider: "mcp"     # baidu / aliyun / tencent / mcp
  mcp:
    url: "http://mcpserver.../sse"
  baidu:
    api_key: "xxx"
    secret_key: "xxx"
  aliyun:
    access_key_id: "xxx"
    access_key_secret: "xxx"
    region: "cn-shanghai"
  tencent:
    secret_id: "xxx"
    secret_key: "xxx"
    region: "ap-guangzhou"

💡 设计亮点：新增 OCR 引擎只需实现 BaseOCR 接口，零侵入现有代码，符合开闭原则

事件含义

📐 自然语言校验规则，配置即生效

config.yaml 定义规则

validation:
  rules:
    - "花费金额校验，金额小于200为通过"
    - "税率必须为 13% 或 6%"
    - "不含税金额 + 税额 = 价税合计"

规则用自然语言描述，由 LLM 理解并执行校验 — 无需写代码

LLM 校验输出 JSON

{
  "results": [
    {
      "rule": "花费金额校验...",
      "field": "价税合计",
      "value": 98.5,
      "passed": true,
      "reason": "金额98.5<200，通过"
    }
  ],
  "summary": {
    "total_rules": 1,
    "passed": 1, "failed": 0
  }
}

🥇

LLM 编排传统工具链

用 Function Calling 让 LLM 自主编排 6 个传统 Tool（DB 查询、文件下载、OCR），而非把 LLM 当作单一问答接口。这是 LLM Agent 的典型落地范式

多 OCR 引擎策略模式

自然语言即规则

SSE 流式 + 进度可视化

6 种 SSE 事件类型，前端实时展示工具执行进度，解决了 LLM 调用长链路中的"黑盒等待"体验痛点

🔬 竞品对比与差异化

维度 🟦 本方案 传统 RPA 通用 OCR 平台
理解能力 LLM 语义理解，非结构化文本灵活提取 规则匹配，格式一变就失效 纯 OCR，不处理语义
规则定义 自然语言规则，业务人员可编写 代码/脚本编写 不支持规则校验
系统集成 原生对接 OA (SQL Server + Cookie) 需定制开发 需手动上传文件
交互方式 自然语言对话 + 流式进度 任务调度触发 API 调用 / Web 上传
OCR 灵活性 4 引擎可切换（内网 MCP / 云 API） 通常绑定单一引擎 绑定自家引擎
部署方式 内网部署，数据不出企业 内网 云端为主
扩展性 新增工具只需添加 Tool + 函数定义 需修改流程脚本 受限于平台功能

🎯 核心差异：我们不是"又一个 OCR 工具"，而是深度融合企业 OA 工作流的 LLM Agent。LLM 不仅是"大脑"，更是"调度中心" — 自主决定调用什么工具、用什么参数、如何解释结果。
